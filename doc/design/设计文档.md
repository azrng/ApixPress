## 1. 架构设计

### 1.1 总体架构

- 技术栈：`.NET 10 + Avalonia 11 + Ursa.Avalonia + CommunityToolkit.Mvvm + Dapper + SQLite`
- 分层结构：`View -> ViewModel -> Service -> Repository -> Model`
- 数据流：
  - 导入 Swagger 文档：`ImportView -> ImportViewModel -> ApiDocumentService -> SwaggerImportService -> Repository`
  - 调试请求：`RequestEditorView -> RequestEditorViewModel -> RequestExecutionService -> HttpClient`
  - 用例保存：`CasePanelView -> CasePanelViewModel -> RequestCaseService -> SQLite Repository`
- 全局消息：
  - 使用 `IMessenger` 分发导航切换、文档导入完成、用例加载、通知消息
- 持久化：
  - SQLite 本地库保存文档、接口定义、环境变量、请求用例、最近使用记录
  - 应用启动时自动初始化数据库与基础表结构

### 1.2 模块划分

| 模块 | 职责 | 说明 |
| ---- | ---- | ---- |
| Shell | 应用主框架、导航、全局状态提示 | 左侧导航 + 顶部工具区 + 主工作区 |
| Import | 文档导入、解析、校验 | 支持本地文件内容导入、URL 导入、拖拽入口占位 |
| ApiExplorer | 接口树、搜索与接口选择 | 树形展示分组与接口，支持关键字过滤 |
| RequestWorkbench | 请求配置与发送 | 编辑方法、URL、参数、Header、Body，发送请求 |
| ResponseViewer | 响应信息展示 | 状态码、耗时、大小、响应头、内容 |
| CaseLibrary | 请求用例保存与复用 | 本地保存、分组、快速加载、复制 |
| Environment | 环境变量管理 | 基础变量增删改查与解析替换 |
| Infrastructure | SQLite、HTTP、配置、初始化 | 数据库初始化、仓储、HTTP 封装 |

### 1.3 关键技术决策

| 决策点 | 方案 | 原因 |
| ---- | ---- | ---- |
| 导航承载 | 单窗口 Shell + 工作台页 | MVP 阶段减少窗口切换复杂度，提升主流程连续性 |
| 文档解析 | 先支持 Swagger/OpenAPI JSON | 与需求范围一致，优先实现主链路 |
| 请求执行 | `HttpClient` + 请求快照 DTO | 便于调试发送和用例复用统一 |
| 数据落库 | SQLite + Dapper | 本地桌面工具最小可用，易于验证真实生效 |
| 初始化 | 首次启动自动建表 | 避免手工迁移门槛，仍保留 SQL 脚本目录 |
| 状态管理 | ViewModel 本地状态 + Messenger 全局消息 | 符合现有规范，减少静态状态污染 |

### 2.2 业务规则

1. 文档导入成功后必须真实持久化，应用重启后仍可看到最近导入的接口文档。
2. 接口树按标签或路径首段分组；无标签时归类到“默认分组”。
3. 选择接口后自动回填方法、URL、路径参数、查询参数、Header 默认值。
4. 发送请求前先解析环境变量占位符，如 `{{baseUrl}}`、`{{token}}`。
5. 请求结果记录状态码、耗时、响应大小、响应头和响应内容。
6. 保存用例时若同名，提示覆盖确认；MVP 允许默认覆盖当前分组下同名用例。
7. 删除用例必须二次确认。
8. 所有新增、编辑、删除操作必须落库生效。

## 4. 交互说明

### 4.1 导入流程

1. 用户点击导入入口，选择 `URL 导入` 或 `本地文件导入`。
2. 系统执行基础校验：
   - 文件大小不超过 20MB
   - 内容必须为合法 JSON
   - 文档至少包含 `paths`
3. 导入成功后保存文档信息、接口、参数定义，并刷新接口树。
4. 若导入失败，在导入面板展示明确原因，并保留重试入口。

### 4.2 调试流程

1. 用户从接口树选择一个接口。
2. 系统自动回填请求方法、URL、默认参数和描述。
3. 用户编辑参数或环境变量引用后点击发送，或使用 `Ctrl+Enter`。
4. 系统组装请求、解析变量、发送 HTTP 请求并记录结果。
5. 响应面板刷新状态码、耗时、大小与响应内容。

### 4.3 用例流程

1. 用户在请求调试成功或编辑完成后点击“保存为用例”。
2. 系统弹出用例信息编辑区域，允许填写名称、分组、标签、描述。
3. 保存成功后，用例列表即时刷新。
4. 点击“加载”会将用例快照回填到请求调试面板；点击“复制”会创建副本。

### 4.4 边界与异常

- 文档为空：工作台展示空状态，禁用发送与保存用例按钮。
- 请求未选择接口：允许手工编辑 URL，但提示当前未绑定接口模板。
- URL 导入失败：保留输入值与错误信息，允许直接重试。
- 请求异常：在响应面板以错误态显示异常摘要，并保留原始请求配置。
- 数据库初始化失败：应用启动弹出阻塞错误提示，不进入主工作台。
- 无权限状态：当前 MVP 为本地单用户，可暂不单独暴露页面，但保留状态位。

### 4.5 当前假设

- 本次实现按需求文档直接进入开发，视为用户已接受本版设计基线。
- MVP 暂不实现团队协作、云同步、复杂脚本和 OpenAPI 3.1/YAML 扩展。

## 5. 多项目与多环境扩展（阶段 0 草案）

> 本章节用于承接“支持多项目管理，并在每个项目下支持多环境 BaseUrl 与变量参数”的新增需求；在用户确认前，仅作为下一阶段开发基线草案。

### 5.1 架构设计

#### 5.1.1 模块增量

| 模块 | 新职责 | 设计说明 |
| ---- | ---- | ---- |
| ProjectWorkspace | 管理项目列表、当前项目选择、默认项目记忆 | 延续单窗口 Shell，不新增独立复杂导航层级 |
| EnvironmentScope | 管理当前项目下的环境列表、激活环境、BaseUrl 和变量 | 环境面板从“单环境变量列表”升级为“环境列表 + 环境详情” |
| WorkspaceIsolation | 为文档、用例、历史记录建立项目隔离 | 减少跨项目串用请求配置的风险 |
| Persistence | 扩展 SQLite 结构与迁移脚本 | 新增项目、项目环境及关联字段 |

#### 5.1.2 数据流

- 项目切换：`MainWindowView -> MainWindowViewModel -> IProjectWorkspaceService -> Repository -> SQLite`
- 环境切换：`EnvironmentPanelView -> EnvironmentPanelViewModel -> IProjectEnvironmentService -> Repository -> SQLite`
- 请求发送：`RequestWorkbenchViewModel -> IRequestExecutionService -> EnvironmentResolver -> HttpClient`
- 数据隔离：当前项目切换后，接口文档、用例、请求历史、环境面板同步按 `ProjectId` 过滤加载

#### 5.1.3 推荐数据模型调整

| 实体/表 | 关键字段 | 用途 |
| ---- | ---- | ---- |
| `projects` | `id`, `name`, `description`, `is_default`, `created_at`, `updated_at` | 项目主表 |
| `project_environments` | `id`, `project_id`, `name`, `base_url`, `is_active`, `sort_order` | 每个项目下的环境定义 |
| `environment_variables` | `id`, `environment_id`, `key`, `value`, `is_enabled` | 变量从按环境名存储升级为按环境实体关联 |
| `api_documents` | 增加 `project_id` | 文档归属项目 |
| `request_cases` | 增加 `project_id` | 用例归属项目 |
| `request_history` | 增加 `project_id` | 历史请求归属项目 |

#### 5.1.4 关键技术决策

| 决策点 | 方案 | 原因 |
| ---- | ---- | ---- |
| 项目承载方式 | 顶部项目切换 + 项目管理侧滑面板/对话框 | 与现有工作台兼容，避免重新拆主框架 |
| 环境承载方式 | 顶部环境下拉 + 右侧环境管理面板 | 当前请求调试场景中切换成本最低 |
| BaseUrl 建模 | 作为项目环境一级字段单独存储 | 避免将核心地址与普通变量混在一起，便于校验和展示 |
| 变量解析顺序 | `BaseUrl` 先参与 URL 组装，再解析自定义变量 | 让请求地址拼装规则更稳定、可预测 |
| 数据迁移策略 | 通过新增迁移脚本兼容旧版 `Default` 环境数据 | 降低已有本地数据升级成本 |
| UI 组件复用 | 优先复用 `E:\Test\Ursa.Avalonia` 中的 `NavMenu`、`Drawer`、`OverlayDialogHost`、`TreeComboBox`、`Form` 等现成能力 | 降低自研控件成本，并保持与 Ursa 生态一致 |

### 5.2 功能需求

#### 5.2.1 页面与入口调整

| 页面/区域 | 新增或调整内容 | 目标 |
| ---- | ---- | ---- |
| 顶部工具栏 | 新增项目切换下拉、项目管理入口、当前环境下拉 | 将工作上下文前置，减少误操作 |
| 左侧侧栏 | 保留请求历史，但按当前项目过滤 | 不扩大无意义层级 |
| 右侧环境面板 | 从变量列表升级为环境列表 + 变量编辑区 | 支持同项目多环境维护 |
| 项目管理面板 | 新增项目新增、重命名、删除、设置默认项目 | 支持多项目管理闭环 |

#### 5.2.2 业务规则

1. 应用启动时优先进入默认项目；若无项目则展示创建项目引导。
2. 每个项目至少保留一个环境，且必须配置可编辑的 `BaseUrl` 字段。
3. 删除当前激活环境前，必须先切换到其他环境或自动指定新的激活环境。
4. 请求发送时仅使用当前项目的激活环境，禁止跨项目读取变量。
5. 当接口路径为相对路径时，使用当前环境 `BaseUrl + Path` 进行最终 URL 组装。
6. 用例、请求历史、已导入文档默认按项目隔离展示。
7. 每个项目默认仅允许一个激活环境，切换环境后新的激活环境立即生效。
8. 删除项目时默认级联清理该项目下的环境、变量、文档、用例和历史记录，并要求二次确认。

#### 5.2.3 关键状态

| 场景 | 状态说明 |
| ---- | ---- |
| 无项目 | 展示项目创建引导，发送按钮置灰 |
| 有项目无环境 | 引导先创建环境，禁止发送请求 |
| 有环境但 `BaseUrl` 为空 | 允许保存草稿，不允许发送相对路径请求 |
| 切换项目中 | 主工作区显示局部 loading，防止旧数据闪现 |
| 删除项目失败 | 停留当前项目并展示失败原因 |

### 5.4 交互说明

#### 5.4.1 项目流程

1. 用户首次进入应用时，若没有项目，界面展示创建项目引导。
2. 创建项目成功后，系统自动创建一个初始环境草稿，可命名为“开发”或“默认环境”。
3. 用户切换项目时，主界面同步刷新该项目的接口文档、用例、历史记录和当前环境。

#### 5.4.2 环境流程

1. 用户在顶部环境下拉中切换环境，当前请求调试上下文立即跟随变化。
2. 用户在环境管理面板中维护 `BaseUrl` 与变量列表，保存后持久化到 SQLite。
3. 若变量被禁用，则在请求发送时不参与占位符替换。

#### 5.4.3 请求发送流程

1. 用户选择项目与环境后编辑请求。
2. 系统先校验当前项目和环境是否完整可用。
3. 对相对路径请求，系统使用环境 `BaseUrl` 补全地址；随后解析其余 `{{key}}` 变量。
4. 请求执行完成后，历史记录写入当前项目作用域。

#### 5.4.4 边界与错误处理

- 项目重名：保存前拦截并提示用户修改名称。
- 环境重名：限制在同一项目内唯一，不影响其他项目。
- 项目删除：二次确认文案必须明确会清理哪些数据。
- 激活环境缺少 `BaseUrl`：发送相对路径请求时报错，并给出前往环境面板的入口。
- 旧数据升级：若本地仅存在旧版 `Default` 环境数据，升级时自动迁移到默认项目的默认环境。

#### 5.4.5 当前假设与待确认

- 当前已确认请求历史、用例、导入文档全部按项目严格隔离。
- 当前已确认每个项目默认只保留一个激活环境；是否需要并排对比多个环境，暂不纳入本阶段。
- 视图开发阶段优先参考 `E:\Test\Ursa.Avalonia` 中现有示例与组件能力实现项目管理、环境管理和抽屉/对话框交互。

## 6. 首页主窗口与项目 Tab 化重构（阶段 0 草案）

| ProjectExplorer | 左侧接口管理导航 | 用于展示项目名、接口管理入口和后续目录树能力 |
| QuickRequestWorkspace | 中央快捷请求工作区 | 默认直接展示可编辑、可发送、带环境变量的快捷请求界面 |
| EnvironmentWorkspace | 环境选择与环境管理弹框 | 顶部窄选择器 + 独立环境管理界面 |

#### 6.1.2 数据流

- 首页打开项目：`HomeWorkspace -> MainWindowViewModel -> OpenProjectTab -> ProjectTabCollection`
- Tab 切换：`ShellTabs -> MainWindowViewModel -> ActiveProjectTab`
- 环境切换：`ProjectWorkspaceTab -> EnvironmentPanelViewModel -> IEnvironmentVariableService`
- 环境管理保存：`EnvironmentManagementDialog -> EnvironmentPanelViewModel -> Service -> Repository`
- 快捷请求：`QuickRequestWorkspace -> MainWindowViewModel -> RequestWorkbench`

#### 6.1.3 关键技术决策

| 决策点 | 方案 | 原因 |
| ---- | ---- | ---- |
| 首页承载方式 | 首页作为固定主 Tab | 保持“项目列表始终可回到”的稳定入口 |
| 项目打开方式 | 点击项目后新开项目 Tab | 更符合多项目并行联调场景 |
| 重复打开项目 | 默认不重复开同一项目 Tab，已打开则切到已有 Tab | 避免 Tab 爆炸与状态重复 |
| 项目页布局 | 左窄导航 + 中央快捷请求编辑区 + 右上环境选择 | 贴近参考图，并让项目打开后立即具备可发送请求的能力 |
| 环境入口 | 项目页右上角窄环境选择框 + 环境设置按钮 | 强化环境切换的高频操作属性 |
| 环境管理布局 | 独立双栏管理界面 | 能承载环境列表、前置 URL 和变量编辑的真实流程 |

### 6.2 功能需求

#### 6.2.1 页面列表

| 页面/区域 | 功能点 | 关键状态 |
| ---- | ---- | ---- |
| 主窗口顶部 Tab 区 | 首页 Tab、项目 Tab、Tab 关闭、Tab 激活 | active / hovered |
| 首页 Tab | 项目列表、新建项目、搜索项目 | empty / list / creating |
| 项目 Tab 工作区 | 项目标题、接口管理入口、快捷请求编辑器、环境选择 | ready / sending / response |
| 环境管理界面 | 环境列表、前置 URL、变量表格、保存 | empty / editing / saving |

#### 6.2.2 业务规则

1. 应用启动默认激活首页 Tab。
2. 首页 Tab 不允许关闭，项目 Tab 允许关闭。
3. 同一个项目若已存在打开中的 Tab，再次点击项目卡片时直接切换到已有 Tab。
4. 每个项目 Tab 必须维持各自独立的当前环境和工作状态。
5. 项目 Tab 中间默认直接展示“快捷请求”编辑器，不再额外放置入口卡片。
6. 环境选择仅影响当前激活项目 Tab，不应串改其他项目 Tab 的上下文。
7. 环境管理中的前置 URL、变量、激活环境修改必须真实落库。
8. 快捷请求必须支持当前环境变量参与 URL、Headers、Params 等字段解析。

### 6.4 交互说明

#### 6.4.1 首页与 Tab 流程

1. 应用启动时默认进入首页 Tab。
2. 用户点击首页中的项目卡片后，系统检查该项目是否已存在打开中的 Tab。
3. 若不存在，则新建项目 Tab 并切换过去；若已存在，则直接激活已有 Tab。
4. 用户关闭项目 Tab 时，仅关闭工作区视图，不删除项目本身。

#### 6.4.2 项目页流程

1. 项目 Tab 激活后，左侧显示该项目的接口管理导航。
2. 中央区域默认直接展示快捷请求编辑器，不再先经过入口卡片页。
3. 快捷请求编辑器保留环境选择、请求地址、参数、Headers、Body、发送与响应查看等核心能力。

#### 6.4.3 环境流程

1. 用户通过项目页右上角环境选择框切换当前环境。
2. 点击“环境设置”打开环境管理界面。
3. 用户可在环境管理界面维护环境列表、前置 URL 和变量，并执行保存。
4. 保存后当前项目 Tab 的环境上下文即时刷新。

#### 6.4.4 待确认与当前假设

- 当前建议首页 Tab 不可关闭，项目 Tab 可关闭。
- 当前建议中间区域首版只放“快捷请求”一个主工作区，不放“新建 HTTP 接口”或其他入口卡片。
- 当前建议环境管理首版只做项目级环境，不实现全局变量、云端 Mock 或密钥库分层。
- 当前建议“项目详情页模仿图 3”的范围限定为信息结构和入口布局，不强求逐像素复刻。

## 7. 关于页检查更新与发布链路

### 7.1 功能目标

- 在设置中心“关于”页提供真实可用的“检查更新”入口，不再停留在 mock 状态。
- 使用主程序内置隐藏更新模式接入桌面端自动更新，避免依赖第三方升级包和独立更新器 exe，并保持发布产物可控。
- 发布时由 GitHub Actions 自动生成更新包与 `versions.json`，客户端固定从 GitHub Releases 最新资产拉取更新清单。

### 7.2 客户端链路

1. 用户在“关于”页点击“检查更新”。
2. `MainWindowViewModel` 调用 `IApplicationUpdateService` 拉取 `versions.json`，解析最新版本信息。
3. 若当前已是最新版本，则在关于页展示检查结果，不启动更新流程。
4. 若发现新版本，则由 `ApplicationUpdateService` 生成更新请求文件并以 `--apply-update --request-file` 参数拉起新的主程序进程。
5. 主程序更新模式自行下载 zip 包、校验 SHA256、解压到临时目录，并通过脚本等待原主程序与更新模式进程退出后替换当前安装目录，再重新启动主程序。

### 7.3 发布与版本清单

- GitHub Actions 发布 `win-x64` 自包含主程序包，并将主程序发布目录压缩为 zip。
- 每次发布同步生成 `versions.json`，字段至少包含：`PacketName`、`Hash(SHA256)`、`Version`、`Url`、`PubTime`。
- 客户端默认清单地址：`https://github.com/azrng/ApixPress/releases/latest/download/versions.json`
- 更新包默认地址：`https://github.com/azrng/ApixPress/releases/latest/download/ApixPress-win-x64-portable.zip`
- 版本号默认在构建与发布时自动按 UTC 时间生成，格式为 `yyyy.M.d.HH`；若根目录 `VERSION` 填写显式版本号（兼容旧机制），则构建与 GitHub Actions 发布优先使用该值；当 `VERSION` 为 `auto`、为空或缺失时，回退到 UTC 日期小时版本。
- 发布目录默认仅保留主程序 `ApixPress.exe`，不再额外生成旧主程序名别名。
