# 修复接口预览递归闪退

## 本次目标

- 修复从左侧接口目录点击接口后，右侧请求编辑区在预览求值阶段触发递归通知并导致应用闪退的问题。

## 核心改动

- 使用 `dotnet-dump` 分析 `ApixPress.exe.31656.dmp`，确认崩溃链路为 `HasResolvedRequestPreview -> BuildResolvedRequestPreviewText -> BuildSnapshot -> RequestName setter -> NotifyStateChanged` 的递归调用。
- 调整 `RequestWorkspaceTabViewModel.BuildSnapshot()`，改为把解析后的请求名作为参数传给 `RequestConfigTabViewModel.BuildRequestSnapshot()`，不再临时写回 `ConfigTab.RequestName`。
- 为 `RequestConfigTabViewModel.BuildRequestSnapshot()` 增加可选的请求名覆盖参数，保持快照构建为纯读取行为。
- 补充回归测试，验证读取 HTTP 接口预览时不会触发 `RequestName` 变更通知。
- 在 `application-AGENTS.md` 增补规则，明确绑定 getter、预览计算和 `Build*Snapshot` 派生方法必须无副作用。

## 回顾分析

- 错误现象：点击接口后应用直接闪退，Windows 事件日志显示 `System.StackOverflowException`。
- 触发场景：请求编辑器绑定读取发送预览文本时，会同步构建请求快照。
- 根因：快照构建为了补齐默认名称，临时写回了 `RequestName`，导致绑定求值期间再次触发属性通知，形成自激递归。
- 本次补充或修改的规则：绑定 getter、预览文本计算与 `Build*Snapshot` 派生方法禁止回写 `ObservableProperty`。
- 后续如何避免或验证：凡是要给预览、文档、curl、生效请求做派生计算时，优先显式传参，不要在 getter 内临时改 ViewModel 状态；新增同类功能时补一条“读取派生属性不触发属性变更”的测试。
